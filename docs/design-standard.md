# 设计规范

## 基本原则

1. 网页工具应以实用为主，先保证信息清晰、操作顺畅，再做视觉美化。
2. 页面布局要适合反复使用，避免只像展示页而不便操作。
3. 视觉风格可以逐步提升，但不能影响功能识别和数据准确性。
4. 所有控件、状态和提示应服务于真实工作流程。
5. 设计修改需要能被当前负责人和用户验收。

## 设计调整汇报要求

每次汇报必须包含：

- 设计范围。
- 调整内容。
- 涉及文件。
- 验证方式。
- 遗留风险或需要用户确认的问题。

## 第一版视觉方向

1. 整体方向为“科技 + 时尚 + 克制高级感”，界面应清爽、安静、可长期使用，避免营销页式大面积装饰。
2. 主色以白色、银灰、浅石墨和深石墨文字为主，少量使用科技蓝或青色强调当前状态、主要操作和拓扑关键路径。
3. 不复制 Apple 品牌、Logo、官方素材、产品页面结构或可识别的品牌表达；只借鉴克制、留白、细腻层次和清晰排版的设计原则。
4. 页面优先服务“输入参数 -> 生成结果 -> 查看清单和拓扑 -> 校验或导出”的工作流程。
5. 装饰元素应少而精，不使用大面积炫光、复杂渐变、过重阴影或影响数据阅读的背景图。

## 视觉语言参考工具

1. 后续需要细化页面视觉时，可以使用本地 `design-md-selector` 技能作为视觉语言参考工具。
2. 第一版按用户明确偏好选择 `apple` 方向作为主要参考，提炼其克制、留白、单一蓝色交互色、系统字体和低阴影层次。
3. 可作为后续方向校准的品牌语言包括 Apple、Vercel、Linear、NVIDIA、x.ai 等偏科技、克制、高级的视觉体系，但每次实际实现前只选择一个最贴近当前页面目标的 `DESIGN.md` 作为主参考，避免混搭失控。
4. 参考内容只用于提炼颜色、排版、间距、控件形态、边框、阴影、响应式节奏和技术感表达，不复制任何品牌 Logo、官方素材、商标、原站页面结构、产品图、图标、标语或文案。
5. 本项目是实用网页工具，不做品牌官网或营销页。任何视觉参考都必须服从参数输入、清单结果、拓扑画布、错误提示和总览信息的清晰可用。

### Apple 方向转译约束

- 画布以白色、浅银灰和近黑文字为主，界面层次来自留白、细边框和浅色表面差异。
- 主交互色统一使用蓝色，接近 Apple Action Blue；青色只用于拓扑辅助链路，不作为第二主色。
- 标题使用系统字体 600 字重，正文保持 17px 左右的舒适阅读节奏；项目遵守全局规则，字间距保持默认。
- 输入控件和主要操作可使用胶囊形态，工作台面板使用较大但克制的圆角。
- 阴影只用于主要面板的轻量层次，不能做厚重卡片阴影、炫光或装饰渐变。
- 不使用 Apple Logo、产品图、官网导航结构、营销文案或可识别页面构图。
- 本项目不是产品发布页，所有视觉处理必须服务于参数输入、清单表格和拓扑可读性。

## 页面结构

第一版页面建议采用单页工作台结构，按用户操作顺序从上到下组织：

1. 顶部状态栏：显示工具名称、当前生成状态、主要操作入口。主要操作包括生成、重置、导出或复制，具体功能入口由当前负责人确认后实现。
2. 参数输入区：放在页面上方，使用分组表单承载项目规模、网络类型、节点数量、设备规格等参数。输入区应支持清晰的必填提示、单位标注和错误反馈。
3. 结果总览区：紧跟输入区，展示服务器、交换机、链路、端口、IP 或其他关键汇总数据。总览卡片数量应克制，优先展示用户判断方案是否合理所需的信息。
4. 四网清单区：以表格或分组表格呈现业务网、存储网、管理网和带外网清单。四网名称应固定可识别，字段对齐一致，便于横向比较。
5. 四网拓扑区：分别展示四类网络的独立拓扑。每个拓扑区应有清晰标题、状态、缩放或查看入口，避免多个拓扑混在一起导致识别困难。
6. 统一大拓扑区：放在四网拓扑之后，用于展示整体关系。该区域应给足画布空间，优先保证节点、链路和分组关系可读。

## 颜色规范

建议第一版使用以下语义，不要求一次性建立复杂主题系统：

- 页面背景：`#F7F8FA` 或接近的冷灰白。
- 主内容背景：`#FFFFFF`。
- 一级文字：`#1D1D1F`，用于标题和关键数据。
- 二级文字：`#42464D`，用于说明、字段名和普通内容。
- 辅助文字：`#6E7480`，用于提示、单位和次要状态。
- 分割线和边框：`#E5E7EB`。
- 主强调色：科技蓝，建议接近 `#2563EB`，用于主按钮、选中态和重点链路。
- 次强调色：青色，建议接近 `#06B6D4`，用于拓扑辅助高亮或网络分类。
- 成功状态：绿色，建议接近 `#16A34A`。
- 警告状态：琥珀色，建议接近 `#D97706`。
- 错误状态：红色，建议接近 `#DC2626`。

强调色使用面积要小，不能让页面变成单一蓝色或高饱和科技风。

## 字体和排版

1. 中文字体优先使用系统字体栈，例如 Windows 下的 Microsoft YaHei UI、Segoe UI 和常见 sans-serif 字体。
2. 页面标题建议 20-24px，区块标题建议 16-18px，正文和表格建议 13-14px，辅助信息建议 12-13px。
3. 字重以 400、500、600 为主，避免过多粗体造成视觉噪音。
4. 字间距保持默认，不使用负字距。
5. 数据、端口、IP、设备型号等可使用等宽字体或等宽数字显示，以提升比对效率。

## 控件规范

1. 输入控件应有明确标签、单位和默认值说明；错误时在控件附近显示原因。
2. 主按钮只保留一个最重要动作，例如“生成”或“重新生成”。次要动作使用弱化样式。
3. 开关类设置使用 toggle 或 checkbox；多选网络类型使用 checkbox；互斥模式使用 segmented control 或 radio。
4. 数字参数优先使用数字输入框，必要时可增加步进器，不默认使用滑块。
5. 表单分组之间保持 16-24px 间距，避免把大量参数堆成密集文本墙。

## 表格规范

1. 四网清单表格应优先保证字段对齐、列宽稳定和横向扫描效率。
2. 表头固定使用浅灰背景或清晰分割线，字段名短而准确。
3. 关键字段如设备名、端口、网络类型、IP 段、连接对象应保持可复制文本。
4. 数量较多时允许表格内部滚动，但不要让整页横向滚动成为主要使用方式。
5. 空数据、生成中、生成失败都应有明确状态，不显示空白区域让用户猜测。

## 拓扑区规范

1. 拓扑图应以“可读”为第一优先级，节点间距、连线方向和网络分组要清楚。
2. 四网拓扑建议使用一致的节点样式，通过标题、标签或小面积色条区分网络类型。
3. 统一大拓扑可以使用分组容器或泳道表达业务网、存储网、管理网和带外网的关系。
4. 节点标签应显示关键名称，详细信息放到 hover、点击详情或侧边信息区中，避免节点本身过载。
5. 线条颜色应克制，重点链路可使用蓝或青色，普通链路使用中性灰。
6. 拓扑画布需要预留缩放、适配视图和导出入口的位置；具体功能由后续实现阶段确认。

## 拓扑专项设计规范

本节用于修正“网页拓扑像流程图”的问题。后续拓扑应以网络工程图为目标：用分区、层级、平面、POD、设备组、服务器网卡条和代表性链路表达结构，而不是用少量圆角节点串成流程。

### 通用拓扑原则

1. 每张拓扑必须先画区域，再画设备，再画链路；区域名称、设备层级和链路类型都应可见。
2. 设备布局应体现网络层级关系，不采用从左到右的流程箭头式排布。
3. 链路只画代表性连接，数量通过标注表达；当设备数量较多时使用省略号和“同模式扩展”说明。
4. 链路线条允许比整体 Apple-inspired 界面更工程化，可使用蓝色虚线分区、设备图标和彩色链路，但画布、文字、面板仍保持白/银/石墨的克制基底。
5. 每张拓扑都需要图例或局部说明，说明颜色、线型、链路速率和省略号含义。
6. 设备型号可显示在节点旁或区块说明中；若型号仍为占位，应避免把占位型号设计成最终品牌物料。

### 计算网独立拓扑

计算网拓扑优先参考“Spine 区 + Leaf 区 + POD 分组 + P1/P2 平面 + 服务器 8 张 CX8 网卡条”的表达方式。

1. 画布上方为 `Spine` 区，使用横向虚线大框，内部展示若干 Spine 设备组。
2. 每个 Spine 设备组必须表达 `P1` 和 `P2` 两个平面，可用左右相邻色块、双节点或半块设备表达；P1/P2 文字必须直接写在设备块内。
3. 画布中下方为 `Leaf` 区，按 POD 拆成独立虚线分组，例如 `POD1`、`POD2`、`PODn`。POD 标签放在分组左下或左上，不能只在说明文字中出现。
4. 每个 POD 内部至少展示首组 Leaf、次组 Leaf、末组 Leaf，并用横向省略号表达中间 Leaf 扩展，例如 `1-1 / 2-1`、`1-2 / 2-2`、`1-8 / 2-8`。
5. Leaf 命名应体现 P1/P2 平面和序号关系。建议 `1-x` 代表 P1 Leaf，`2-x` 代表 P2 Leaf；颜色可用红/橙或蓝/青双平面，但同一画布必须一致。
6. 服务器应以“网卡条 + 服务器底座”表达，服务器上方展示 8 张 CX8 网卡竖条，并标注 `1` 到 `8`。网卡条颜色可用多色区分端口，但饱和度要控制，避免盖过链路。
7. 服务器底座显示服务器序号或范围，例如 `1`、`2`、`32`、`97`、`98`、`128`。当服务器数量多时，只展示首尾和代表节点，中间使用省略号。
8. Spine 到 Leaf、Leaf 到服务器的连线应表现为多对多关系，不能只画单线串联。为避免过密，只画代表性连线，并用文字说明“省略号表示同模式扩展”。
9. 画布底部或右下角必须标注关键数量：`B300 台数`、`POD 数`、`Leaf 数`、`Spine 数`、`每台 8*CX8`、`P1/P2 双平面`。
10. B300 超出当前二层方案支持范围时，不显示普通流程图，应显示清晰的“不生成当前二层计算网拓扑 / 后续支持三层组网”状态。

### 存储网拓扑

1. 存储网应展示 `Spine 层 -> Leaf 层 -> 服务器接入层` 的三层关系。
2. Spine 和 Leaf 使用独立横向分区；服务器接入层至少区分 `GPU 服务器`、`全闪服务器`，如后续需要可增加 `混闪服务器`。
3. 400G 连接使用统一颜色，建议采用青色或蓝青色实线，并在图例中标注 `400GE`。
4. 服务器侧标注应同时显示服务器数量和 400G 端口贡献，例如 `GPU 服务器 128 台 / 256 个 400G`。
5. 当 Spine、Leaf 或服务器组数量较多时，只画 2-4 个代表设备，使用 `...` 和数量标签表达扩展。
6. 存储网不能画成“GPU -> 存储 -> 交换机”的流程，应体现 GPU 与存储服务器共同接入存储 Leaf，并向上汇聚到存储 Spine。

### 带内管理网拓扑

1. 带内管理网应展示 `安全出口区 -> Border/Core -> Leaf -> 服务器 25G 接入` 的层次。
2. 安全出口区应包含边界交换机、防火墙等代表设备；若有安全管理交换机，应放入安全设备区而不是混入服务器接入层。
3. Border、Core、Leaf 可以在同一大分区内分层排布，但名称要清楚，不能只显示“管理网交换机”。
4. 服务器接入层应表达全部服务器或服务器池通过 `25GE` 接入带内管理 Leaf，线条建议使用绿色，并在图例中标注。
5. 若存在 100GE 上联或安全出口链路，应与 25GE 接入链路区分颜色或线宽。
6. 带内管理网应强调“业务管理/系统管理流量”，不要与带外 BMC 管理口混淆。

### 带外管理网拓扑

1. 带外管理网应展示 `OOB 汇聚 -> OOB 接入 -> 服务器 BMC / 网络设备管理口` 的层次。
2. OOB 汇聚和 OOB 接入分成两个横向区块，设备数量通过代表节点和数量标签表达。
3. 服务器侧应明确标注 `BMC` 或 `服务器管理口`；网络设备侧应明确标注 `交换机/防火墙管理口`。
4. 带外链路建议使用红色或红橙色细线，并在图例中标注 `GE` 或具体速率。
5. 带外拓扑不能只连接服务器池，还必须表达网络设备和安全设备的管理口也接入 OOB。
6. 如带外口数量由计算网、存储网、带内网设备数量推导，应在拓扑说明中标注“网络和安全设备管理口按当前清单推导”。

### 统一四网拓扑

统一四网拓扑应参考“全局分区网络工程图”的表达：用大区域捋清四张网和安全出口的相对关系，而不是把四张网做成四个流程节点。

1. 建议画布采用纵向分区：
   - 顶部：`安全出口区`，包含边界交换机、防火墙、安全管理设备等。
   - 中部左侧：`带外管理网`，包含 OOB 汇聚、OOB 接入、服务器 BMC 和网络设备管理口。
   - 中部右侧：`存储网 / 带内管理网`，包含存储 Spine/Leaf、带内 Border/Core/Leaf、服务器池。
   - 底部：`计算网 ROCE`，包含 P1/P2 Leaf、Spine、POD 或代表服务器接入关系。
2. 四网区域必须用浅色底或虚线边界区分，区域标题应固定出现：`计算网`、`存储网`、`带内管理网`、`带外管理网`、`安全出口区`。
3. 统一拓扑必须有颜色/线型图例，至少区分 `GE`、`25GE`、`100GE`、`200GE`、`400GE/800GE`。如实际阶段没有某类链路，可以在图例中隐藏或置灰。
4. 线条颜色建议：
   - GE / 带外：红色或红橙色。
   - 25GE / 带内接入：绿色。
   - 100GE / 出口或上联：橙色。
   - 200GE：蓝色。
   - 400GE / 800GE / 计算或存储高速链路：深绿、蓝青或醒目的工程色。
5. 服务器池应位于统一拓扑中部，作为四网共同接入对象，至少区分 `GPU 服务器池`、`存储服务器池`、`管理服务器池`。
6. 统一拓扑要表达关系，而不是穷举所有连线。应画代表性链路，并用区域说明和数量标注补足。
7. 计算网区域不能被挤成一个小节点；它应保留 P1/P2、Spine/Leaf 或 POD 的关键结构，否则统一拓扑无法说明计算网关系。
8. 安全出口区应与带内管理网、必要的带外管理链路建立关系；不应直接把所有网络都画成通向防火墙的单向流程。
9. 统一拓扑底部可增加服务器网卡说明框，列出 GPU、全闪、管理服务器的典型网卡配置，帮助用户理解四网分别使用哪些端口。

### 拓扑响应式策略

1. 拓扑画布桌面优先，独立拓扑建议最小逻辑宽度不低于 1100px，统一四网拓扑建议最小逻辑宽度不低于 1280px。
2. 桌面端优先保证完整分区、图例、数量标注和代表性链路可见。
3. 中小屏幕允许拓扑容器横向滚动，不能压缩到文字重叠、设备不可读或链路遮挡标题。
4. 移动端可提供缩略查看，点击后进入全宽或横向滚动查看；但缩略状态仍要显示拓扑标题、区域名称和图例入口。
5. 拓扑中的文字、节点、图例和线条不得因响应式缩放互相遮挡；若空间不足，应减少代表节点数量，而不是缩小到不可读。

### 拓扑验收点

以下条件满足时，独立拓扑才算“不像流程图”：

1. 能一眼看出网络层级，例如 Spine、Leaf、POD、服务器接入层，而不是一串节点顺序连接。
2. 能看出 P1/P2 或不同平面的关系，特别是计算网必须显示双平面。
3. 能看出服务器侧网卡或接入口如何接入网络，计算网必须显示 8 张 CX8 网卡条或等效表达。
4. 多设备、多服务器通过代表节点、省略号和数量标注表达，而不是省略成一个抽象节点。
5. 链路颜色、线型或图例能说明链路类型和速率。

以下条件满足时，统一四网拓扑才算“四网关系清楚”：

1. 四张网和安全出口区都有独立区域，区域边界和标题明确。
2. 服务器池作为四网共同接入对象可见，能看出 GPU、存储、管理服务器分别连接哪些网络。
3. 带外管理网明确连接服务器 BMC 和网络/安全设备管理口。
4. 带内管理网明确通过 Border/Core/Leaf 与安全出口或管理流量相关区域连接。
5. 存储网和计算网都保留自己的高速网络层级，不被简化成普通流程节点。
6. 图例能解释不同颜色和链路速率，用户无需猜测红、绿、蓝、橙线分别代表什么。

### 本轮浏览器批注整改规范

本节针对浏览器验收批注补充小范围整改要求。整改目标是让拓扑完整可见、可操作、连线规整、图标统一，并修正统一四网拓扑中的安全链路层级。该整改仍只属于拓扑呈现范围，不扩展到导出、保存、账号或其他一阶段外功能。

#### 视口、缩放和可操作策略

1. 每张独立拓扑和统一四网拓扑都必须放入明确的拓扑视口容器，容器要么完整适配 SVG，要么提供横向滚动或拖拽平移能力，不能让右侧内容被裁掉且无查看方式。
2. 桌面端优先采用“容器内完整可见 + 必要时横向滚动”的方案；如果实际 SVG 逻辑宽度大于容器宽度，容器必须显示滚动条或提供明确的拖拽提示。
3. 统一四网拓扑和复杂计算网可以保留较大逻辑画布，但默认首屏必须显示左边界、右边界、标题、图例和主要区域标题，用户能通过滚动或拖拽看到全部内容。
4. 不允许通过过度缩小字体、节点或线条来强行塞进容器。若空间不足，优先减少代表设备数量或增加横向滚动。
5. 移动端允许缩略预览，但必须保留标题、区域边界和查看方式；缩略图不能遮挡或截断关键标签。

#### 统一图标和节点风格

1. 四张独立拓扑和统一四网拓扑应使用同一套设备图标语言：交换机、防火墙、服务器、服务器网卡条、OOB 接入设备的形态要一致。
2. 计算网、存储网、带内管理网、带外管理网可以用不同颜色区分链路和区域，但设备图标的圆角、描边、字号、标签位置和阴影层级应统一。
3. 服务器节点必须采用同一套服务器表达方式。计算网展示 8 张 CX8 网卡条；其他网络若不展示网卡条，也应使用一致的服务器池卡片，不要混用完全不同的图标风格。
4. 同一页面内不要同时出现“工程网络图标”和“普通流程图圆角块”两套节点语言。

#### 规整连线策略

1. 链路应采用分层、分组和少量弯折的方式布线，优先使用水平/垂直/斜向短线组合，避免大量无规则交叉线。
2. 同一层级之间的多条链路应共享起止锚点或使用总线式汇聚线，再分叉到目标设备；不要让每台设备随意向所有设备拉散线。
3. 代表性链路必须足够表达网络关系：Spine 到 Leaf、Leaf 到服务器、OOB 接入到 BMC、带内 Leaf 到服务器都必须有可见连线。
4. 连线不能穿过关键文字、图例、设备标签和区域标题。若无法避免，应调整节点位置、改用折线或减少代表性连线数量。
5. 每类链路颜色必须与图例一致，线宽和透明度保持稳定。普通代表线可细，主干链路可稍粗，但不要用过粗线条压住设备。

#### 计算网整改画法

1. 计算网必须按参考图式布局：上方独立 `Spine` 区，下方按 POD 拆分多个独立框，每个 POD 框内包含 Leaf 和服务器代表节点。
2. `POD1`、`POD2`、`PODn` 必须各自有虚线或浅色边界框，不能只用一个大 Leaf 区粗略覆盖所有 POD。当前实现要求 `pods <= 5` 时完整展示所有 POD，B300=138 时必须显示 POD1~POD5。
3. 每个 POD 内必须展示 P1/P2 Leaf 对，例如 `P1-L1 / P2-L1`、`P1-L8 / P2-L8`、`P1-L16 / P2-L16`，并用省略号表示中间同构设备。
4. Leaf 与 Spine 之间必须有代表性多对多连线，Server 与 Leaf 之间也必须有代表性连线。缺任一层连线即不通过。
5. 服务器仍按“8 张 CX8 网卡条 + 服务器底座”表达，至少展示每个代表 POD 的首台、次台和末台服务器或数量范围。物理双平面末组不足 32 台时，图中必须标注“末组不足 32 台，Leaf 仍按 16 台配置”。
6. 物理双平面计算网应补充 Spine 逃生链路的视觉标注，说明双平面对应 Spine 间存在 8×400G 逃生链路。

#### 存储网整改画法

1. 存储网继续采用 `Spine 层 -> Leaf 层 -> 服务器接入层`，但图标风格必须与计算网一致。
2. Spine 与 Leaf 不强制穷举所有设备，优先使用 `Spine1 / Spine2 / Spine… / SpineN`、`Leaf1 / Leaf2 / Leaf… / LeafN` 的代表节点表达总体数量。
3. 400G 链路应按层级规整连接：Spine 到 Leaf 使用少量主干代表线，Leaf 到 GPU / 全闪服务器池使用分组代表线。
4. GPU、全闪服务器应分组展示，并直接标注各自 400G 接入口贡献；当前一阶段存储网不接入混闪服务器，混闪不应画入存储网接入区。
5. 存储网必须完整可见或可横向滚动查看，右侧设备和说明不能被裁掉。
6. 连线不得密集覆盖设备标签；必要时使用总线式汇聚线和少量分叉线表达代表关系。

#### 带内管理网整改画法

1. 带内管理网必须按 `安全出口 -> Border -> Core -> Leaf -> 服务器接入` 的层级自上而下或由上到下分区表达。
2. 服务器接入层必须分别罗列 `GPU 服务器`、`全闪服务器`、`混闪服务器`、`管理服务器` 四类服务器池，不能只展示“全部服务器”一个汇总节点。
3. 25GE 接入线应从带内 Leaf 规整连接到各服务器池；100GE 或上联线应与 25GE 线区分颜色或线宽。
4. 防火墙、Border、Core、Leaf 的链路关系必须可读，不能用交叉线堆叠在同一区域内。
5. 带内拓扑必须完整可见或可操作查看，不能右侧遮挡且无拖拽/滚动能力。

#### 带外管理网整改画法

1. 带外管理网必须按 `OOB 汇聚 -> OOB 接入 -> 管理对象` 表达，管理对象至少包括服务器 BMC 和网络/安全设备管理口。
2. 服务器管理对象应按 `GPU 服务器 BMC`、`全闪服务器 BMC`、`混闪服务器 BMC`、`管理服务器 BMC` 分组或分列展示，避免只画一个笼统服务器节点。
3. 网络设备管理口和安全设备管理口应单独成组，表达它们也接入 OOB。
4. OOB 链路应采用规整的红色或红橙色线条，从 OOB 接入层分组连接到管理对象，避免散乱交叉。
5. 带外拓扑必须完整可见或可操作查看，右侧内容和设备标签不能被裁掉。

#### 统一四网安全链路层级整改

统一四网拓扑中的安全链路必须按以下垂直层级表达：

1. 顶部为 `边界交换机`。
2. `边界交换机` 串联连接到下方 `防火墙`。
3. `防火墙` 串联连接到下方 `Border`。
4. `Border` 向下连接到 `带内核心`。
5. `带内核心` 再向下连接到 `带内接入 Leaf`。
6. `带内接入 Leaf` 再连接到 GPU、全闪、混闪、管理服务器的带内管理口。

统一四网拓扑不得把边界交换机、防火墙、Border、带内核心、带内接入画成并列节点或随意互连。安全出口到带内网络必须是清晰串联主链路，再从带内接入层分发到服务器池。

#### 本轮整改验收标准

满足以下条件才算“完整可见/可操作”：

1. 桌面端打开页面后，每张拓扑的标题、左边界、右边界、图例和主要设备区都可通过完整适配、横向滚动或拖拽查看。
2. 任一拓扑不允许出现右侧内容被裁切且没有滚动条、拖拽或缩略查看入口。
3. 移动端可以横向滚动，但不能出现文字、按钮、图例或节点互相遮挡到无法识别。

满足以下条件才算“连线不混乱”：

1. 用户能看出每类链路从哪一层到哪一层，不需要猜测线条起止。
2. 主要文字、区域标题和设备标签不被连线覆盖。
3. 同一链路类型颜色一致，且图例解释清楚。
4. 代表性连线表达结构即可，不追求穷举所有物理连线。

满足以下条件才算“四网关系正确”：

1. 统一四网区域边界清楚，计算网、存储网、带内管理网、带外管理网、安全出口区各自可识别。
2. 安全链路严格按 `边界交换机 -> 防火墙 -> Border -> 带内核心 -> 带内接入` 串联表达。
3. 带外管理网单独表达 BMC 和网络/安全设备管理口，不与带内管理网混线。
4. 存储网、计算网保留各自 Spine/Leaf 或 P1/P2/POD 的高速网络结构。
5. GPU、全闪、混闪、管理服务器在统一拓扑中能看出分别接入哪些网络。

## 统一大拓扑布局与连线原则补充（2026-07-03）

### 区域布局

1. 统一大拓扑固定为五区加一条服务器源区：
   - 顶部中间：`安全出口区`
   - 中上整行：`服务器源区`
   - 下方左：`计算网`
   - 下方中：`存储网`
   - 中右：`带内管理网`
   - 下方右：`带外管理网`
2. 服务器源区必须处于计算网、存储网上方，视觉上明确表达“计算/存储从服务器下侧网卡向下接入对应 Leaf”。
3. 安全出口区只和带内主链路发生关系，不与计算网、存储网、带外管理网直接相连。

### 服务器参与边界

1. `B300 GPU 服务器` 参与：计算网、存储网、带内管理网、带外管理网。
2. `全闪服务器` 参与：存储网、带内管理网、带外管理网。
3. `混闪服务器` 参与：带内管理网、带外管理网。
4. `管理服务器` 参与：带内管理网、带外管理网。
5. `网络/安全设备管理口` 只属于带外管理网的管理对象，不属于服务器源区分组。

### 连线原则

1. 计算网链路只允许从 `B300 GPU 服务器` 下侧计算网卡向下连接到计算网 Leaf。
2. 存储网链路只允许从 `B300 GPU 服务器` 和 `全闪服务器` 下侧存储网卡向下连接到存储网 Leaf。
3. 带内管理网链路允许从四类服务器分组分别连接到带内管理区，再通过 `Leaf -> Core -> Border -> 安全出口区` 串联；`Core1/Core2` 必须画在带内管理网区域内。
4. 带外管理网链路允许从四类服务器分组的 `BMC` 连接到下方右侧 OOB 区；`网络/安全设备管理口` 作为 OOB 区内部管理对象展示，不从服务器源区出发。
5. 连线继续采用代表性规整折线，避免长横线穿越多个区域，避免覆盖区域标题和设备标签。

### 状态占位

1. `storageEnabled=false` 时，存储网区域保留标题与说明框，显示 `未配置 400G 存储网卡`，不画假设备。
2. `computeSupported=false` 时，计算网区域保留标题与说明框，显示 `二层计算网本阶段不生成`，不画错误结构。
3. 数量为 `0` 的服务器分组不渲染，避免出现空分组卡片或悬空链路。

## 统一大拓扑参考图对齐补充（2026-07-03）

### 版式

1. 本图按用户提供的参考图直接排版，不再追求抽象概览式均分布局。
2. 安全出口区框外不再保留顶部蓝色横线或 `ISP` 云，避免把框外装饰误读为拓扑组成部分。
3. 左上为图例框；上中偏左为安全出口区；中右为带内管理网；中上为服务器源区；服务器下方左侧为计算网、中部为存储网、右侧为带外管理网。

### 服务器源区

1. 每个服务器分组都要画出网卡条带，再画服务器机身，不允许只画服务器名称后直接起线。
2. 网卡条带按网络类型拆分，并显示数量；颜色与对应网络链路颜色一致。
3. 计算网卡、存储网卡放在服务器卡片下侧，链路从下侧网卡向下接对应网络 Leaf，形成“服务器下侧端口出线”的第一视觉判断。
4. 带外管理网卡与计算/存储网卡同样放在服务器卡片下侧，向右下方接入 OOB 接入层；带内管理网卡可放在服务器卡片上侧，用于减少与下方高速网络链路的交叉。
5. 服务器机身可保留灰色简化矩形，不引入品牌贴图、背景材质或供应商标识。
6. 当全闪服务器和混闪服务器均为 0 时，删除服务器源区左侧的“参数导入、参数存储”辅助弧线和文字。
7. 当计算网 `Leaf=0` 且只剩 Spine 时，计算网区域内 Spine 节点上移到原 Leaf 的视觉高度，并从 B300 计算网卡直接连到 Spine 做示意。
8. 安全出口区标题采用顶部横排文字，不再使用右侧竖排标题。
9. 带内 Core 到两台 Border 必须表达双归关系：每台 Core 分别连接两台 Border，统一图中共 4 条代表链路；链路应分散落到两台 Border 的下联口和 Core 的上联口，不堆在 Core 标签上。
10. 带内管理网的数量说明应放在绿色区域右上角，不能被 Core 节点或 Border-Core 线遮挡。
11. 服务器到带内管理网的来源线应从服务器上侧带内网卡先上绕，再接入带内 Leaf，不能横向穿过服务器卡片或关键标签。
12. 当全闪服务器数量为 `0` 时，删除服务器源区左侧的“参数导入、参数存储”辅助弧线和文字，即使混闪服务器仍存在也不保留该辅助说明。

### 取舍

1. 本轮允许借鉴参考图的布局、分区、相对位置和链路表达方式。
2. 本轮不引入参考图中的品牌 Logo、背景底纹和厂商命名。
3. 当解释文字与链路、节点发生冲突时，优先删减解释文字，保证主图清晰。

## 错误提示和状态

1. 表单错误应就近提示，说明用户需要修改什么。
2. 全局错误应放在输入区或结果区上方，语气直接，不使用模糊文案。
3. 生成中状态需要显示明确进度感，例如按钮 loading、结果区占位或拓扑区骨架。
4. 无结果状态应说明当前需要先输入参数并生成，不展示空表格或空拓扑。
5. 错误色只用于真实错误，警告色用于可继续但需要注意的参数或结果。

## 响应式要求

1. 桌面端优先，建议主内容宽度在 1200-1440px 范围内自适应。
2. 宽屏下可采用“参数输入 + 结果总览”双列或分栏布局；中等宽度改为上下布局。
3. 移动端至少保证参数录入、结果总览和表格查看可用；复杂拓扑允许提示横向查看或进入全屏查看。
4. 控件、卡片、表格和拓扑区要有稳定尺寸，避免加载状态或长文本导致布局跳动。
5. 任何宽度下，按钮文字、表头、节点标签和错误提示都不能互相遮挡。

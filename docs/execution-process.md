# 执行步骤

## 标准流程

1. 用户提出需求。
2. 识别目标、约束、风险和最小可行改动范围，形成小阶段方案。
3. 需求明确且风险可控时，按最小范围实现；高风险操作必须先说明风险并等待用户确认。
4. 实现完成后，必须自测并保留可核查结果。
5. 检查文件、运行结果和自测证据。
6. 向用户汇总验收情况。
7. 用户批准验收通过后，当前阶段结束。
8. 如验收不通过，明确整改项、影响范围和验证方式，修正后重新验证。

## 规则类需求执行流程

涉及计算规则、清单公式、拓扑数量、设计规范或验收口径变化时，必须按以下顺序处理：

1. 先确认用户给出的规则范围，例如本轮规则是否只适用于 B300=5-128，是否影响 129-224。
2. 先修改计算源头，优先检查 `src/lib/calculate.ts`。
3. 再修改展示联动，确保拓扑组件读取或复用同一套派生结果，避免清单与拓扑不一致。
4. 再修改表格计算依据文案，文案必须直接说明对应数量字段的来源。
5. 再补充或修订单元测试，覆盖边界值和用户点名问题。
6. 最后同步 `docs/requirements.md`、`docs/technical-standard.md`、`docs/design-standard.md`、`docs/compute-network-rules.md`、`docs/acceptance-standard.md`，必要时同步 `AGENTS.md`。

## 计算网规则变更专项检查

每次涉及计算网 B300/POD/Leaf/Spine 规则时，必须检查：

- B300=1-4 是否仍走双 Spine 直连，Leaf=0。
- B300=5-128 是否仍走虚拟双平面。
- B300=129-224 是否仍走物理双平面。
- 单 POD Leaf 是否不超过 16，单平面 Leaf 是否不超过 8。
- 清单 Leaf 总数、拓扑 Leaf 分布、底部摘要 Leaf 数是否一致。
- `计算网 LEAF 交换机` 计算依据是否以 Leaf 为主语，避免只写 POD 逻辑导致用户误读。

## 跟踪状态

当前阶段必须维护清晰状态：

- `进行中`：当前阶段正在实现或整理。
- `待验证`：实现已完成，正在自测或复核。
- `整改中`：验证未通过，正在修正。
- `待用户审批`：已完成验收汇总，等待用户最终确认。

不得让阶段无状态停留；必须继续实现、验证、整改或汇报用户。

## 执行边界

1. 每轮只修改与当前需求直接相关的功能代码、页面样式、计算逻辑、构建配置或文档内容。
2. 可以按需更新 `AGENTS.md`、执行流程、验收标准和开发日志；涉及技术路线或设计规则变化时，先更新对应规范文件。
3. 用户提出新需求后，先形成小阶段计划；需求明确且风险可控时直接实现。
4. 只有用户批准后，阶段结束。
5. 直接修改实现文件后，必须说明修改范围、自测方式、自测结果和遗留风险。

## 每日记录

每天首次开展项目工作时，创建或更新 `development-logs/YYYY-MM-DD.md`，记录：

- 今日完成。
- 今日待办。
- 验证记录。
- 风险和问题。

## 推进限制

每个阶段只处理一组相关需求。不得在未完成验收前扩大到新的功能范围。

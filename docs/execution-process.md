# 执行步骤

## 标准流程

当前最高授权：当前会话负责人是项目最高负责人，可以亲自修改任意必要文件并完成验证。

1. 用户向当前负责人提出需求。
2. 当前负责人澄清关键点，形成小阶段方案。
3. 当需求明确且风险可控时，当前负责人直接实现。
4. 实现完成后，当前负责人必须自测并保留可核查结果。
5. 当前负责人检查文件、运行结果和自测证据。
6. 当前负责人向用户汇总验收情况。
7. 用户批准验收通过后，当前阶段结束。
8. 如验收不通过，当前负责人直接整改并重新验证。

## 跟踪状态

当前负责人必须为当前阶段维护清晰状态：

- `进行中`：当前阶段正在实现或整理。
- `待验证`：实现已完成，正在自测或复核。
- `整改中`：验证未通过，正在修正。
- `待用户审批`：当前负责人已完成验收，等待用户最终确认。

当前负责人不得让阶段无状态停留；必须继续实现、验证、整改或汇报用户。

## 当前负责人执行边界

1. 当前负责人可以亲自改功能代码、页面样式、计算逻辑或构建配置。
2. 当前负责人可以直接更新 `AGENTS.md`、需求、执行流程、验收标准和开发日志。
3. 用户提出新需求后，当前负责人先形成小阶段计划；需求明确时直接实现。
4. 只有当前负责人验收通过后，才向用户汇报验收结果；只有用户批准后，阶段结束。
5. 当前负责人直接修改实现文件后，必须说明修改范围、自测方式、自测结果和遗留风险。

## 每日记录

每天首次开展项目工作时，创建或更新 `development-logs/YYYY-MM-DD.md`，记录：

- 今日完成。
- 今日待办。
- 验证记录。
- 风险和问题。

## 推进限制

每个阶段只处理一组相关需求。不得在未完成验收前扩大到新的功能范围。
